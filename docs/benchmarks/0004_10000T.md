目的

10000ターンで実運用規模での安定性を確認

結果

Recall Precisionは約0.93を維持。

Prompt長は平均約430トークンで安定。

ThreadGroup数も平均約2件とほぼ一定。

Conversation Fatigueも継続して機能。

Working Memoryサイズは会話長に依存せず安定した。

考察

Conversationが10000ターンまで増加しても、

Promptサイズはほぼ増加しなかった。

ThreadGroup・Activation Gate・Conversation Fatigueにより、

Working Memoryへ必要なTraceだけを選択できていることを確認した。

今回の評価により、

Trace Recall Engineは会話履歴の長さではなく、

Working Memoryの大きさでスケールする構造であることが確認できた。

設計は10000ターン規模でも破綻しておらず、

今後の研究対象は保存方式ではなく、

Recall Selection・Working Memory Policy・Conversation Policyへ移行する。
