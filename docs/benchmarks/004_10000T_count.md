# Benchmark 004 - 10000T Count

## Purpose

10000ターンでTrace Recall Engineが実運用規模でも安定するか確認する。

## Configuration

| Item | Value |
|---|---|
| Conversation | long_10000t_trace_recall_stress.jsonl |
| DB | eval_long10000_count.db |
| Storage Mode | count |
| Weighted Mode | excluded |
| Response Generation | skipped by --no-response |

## Results

| Metric | Value |
|---|---:|
| Evaluation Turns | 1424 |
| Expected Hits | 1802 |
| Unexpected Hits | 109 |
| Average Precision | 0.927 |
| Fatigue Suppressed | 764 |
| Average Prompt Tokens | 429 |
| Max Prompt Tokens | 693 |
| Average Thread Groups | 1.78 |
| Max Thread Groups | 4 |

Response Generation は `--no-response` により skipped であり、response quality は評価していない。

## Findings

10000ターン規模でもRecall Precisionは高水準を維持した。

Prompt長は平均約430トークンで安定し、最大でも約700トークン未満だった。

ThreadGroup数も平均約2件、最大4件に収まった。

Conversation Fatigueも継続して機能した。

## Interpretation

Conversationが10000ターンまで増加しても、Promptサイズはほぼ増加しなかった。

ThreadGroup・Activation Gate・Conversation Fatigueにより、Working Memoryへ必要なTraceだけを選択できている。

Trace Recall Engineは会話履歴の長さではなく、Working Memoryの大きさでスケールする構造であることを確認した。

## Conclusion

Trace Recall Engineの基礎設計は10000ターン規模でも破綻しなかった。

今後の研究対象は保存方式ではなく、Recall Selection・Working Memory Policy・Conversation Policyへ移行する。
