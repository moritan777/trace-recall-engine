# Benchmark 002 - 1000T Count

## Purpose

Count方式、ThreadGroup、Activation Gate、Conversation Fatigueを含むTrace Recall Engineが、1000ターン規模で破綻しないか確認する。

## Configuration

| Item | Value |
|---|---|
| Conversation | long_1000t_trace_recall_stress_v2.jsonl |
| DB | eval_long1000_count.db |
| Storage Mode | count |
| Weighted Mode | excluded |
| Response Generation | skipped by --no-response |

## Results

1000T評価では、Recall Precisionは高水準を維持し、Promptサイズも増加しなかった。

評価ターン数は143。

Working Memoryへ入ったThreadGroupは合計306、Wordは570。

Fatigueによる抑制は54回。

Response Generation は `--no-response` により skipped であり、response quality は評価していない。

## Findings

1000ターンでもTrace Recall Engineは破綻しなかった。

ThreadGroupにより、保存されたTrace数が増えてもPromptへ渡す情報量を抑制できた。

Activation GateとFatigueにより、「思い出す」と「話す」を分離できた。

## Interpretation

1000T評価により、ボトルネックは保存方式ではなくRecall Selectionであることが明確になった。

この段階で研究テーマは、

Memory Storage

から

Recall Selection / Working Memory

へ移行した。

## Conclusion

Count方式を継続し、Weighted方式は今後の主要評価対象外とする。
