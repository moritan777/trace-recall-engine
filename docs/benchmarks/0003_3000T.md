# Benchmark 003 - 3000T Count

## Purpose

3000ターンでTrace Recall Engineのスケーラビリティを確認する。

## Configuration

| Item | Value |
|---|---|
| Conversation | long_3000t_trace_recall_stress.jsonl |
| DB | eval_long3000_count.db |
| Storage Mode | count |
| Weighted Mode | excluded |
| Response Generation | skipped by --no-response |

## Results

Precisionは1000Tとほぼ同等。

Prompt長もほぼ変化なし。

ThreadGroup数も安定。

失敗は周期的に発生。

Response Generation は `--no-response` により skipped であり、response quality は評価していない。

## Findings

Recall Engineは会話長の増加によって劣化しているわけではなかった。

失敗はランダムではなく、評価シナリオの周期で同じ種類の難問に遭遇している可能性が高い。

Promptサイズ、ThreadGroup数、Working Memoryサイズはいずれも安定していた。

## Interpretation

3000TでもTrace Recall Engineは破綻しなかった。

この結果は、Trace Recall EngineがConversation全体をPromptへ持ち込むのではなく、Working Memoryへ必要なTraceだけを載せる構造として機能していることを示している。

## Conclusion

Count方式の長期スケーラビリティを確認し、今後の評価対象はRecall SelectionとWorking Memory Policyへ移行する。
